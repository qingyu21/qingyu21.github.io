1. Getting Started

1.1 一条入门命令

docker run -d -p 80:80 docker/getting-started
  • -d 在detached（in the background）模式下运行容器
  • -p 80:80 将主机的80端口映射到容器中的80端口
  • docker/getting-started 需要使用的容器镜像

1.2 什么是容器？

容器是你机器上的一个沙盒进程，与主机上的所有其他进程隔离。

That isolation leverages kernel namespaces and cgroups, features that have been in Linux for a long time.

1.3 什么是镜像？

当运行一个容器时，它使用一个隔离的文件系统。这个自定义的文件系统是由一个容器镜像提供的。

由于镜像包含了容器的文件系统，它必须包含运行一个应用程序所需的一切–所有的依赖关系、配置、脚本、二进制文件等。镜像还包含容器的其他配置，如环境变量、运行的默认命令和其他元数据。

2. Sample Application

2.1 获取源码

https://github.com/docker/getting-started/tree/master/app

2.2 构建容器镜像

1. Dockerfile

需要编写Dockerfile文件，它是一个基于文本的指令脚本，用来创建一个容器镜像。

# syntax=docker/dockerfile:1
FROM node:12-alpine
RUN sed -i 's/dl-cdn.alpinelinux.org/mirrors.tuna.tsinghua.edu.cn/g' /etc/apk/repositories
RUN apk add --no-cache python2 g++ make
WORKDIR /app
COPY . .
RUN yarn install --production
CMD ["node", "src/index.js"]
EXPOSE 3000
  • FROM 指定了base镜像
  • RUN 在不调用bash的情况下执行命令
  • WORKDIR 为后面的任何ADD、COPY、CMD、ENTRYPOINT或RUN指令设置工作目录，如果它不存在，将被创建。
  • COPY 将新的文件和目录复制到image的文件系统中
  • CMD 为这个执行中的容器设置默认的可执行文件和参数
  • EXPOSE 定义该容器在运行时要监听的网络端口

2. build

docker build -t getting-started .
  • -t 表示对镜像进行命名
  • . 告诉Docker应该在当前目录下寻找Dockerfile

2.3 启动一个容器

docker run -dp 3000:3000 getting-started

3. Update the application

3.1 Update the source code

如果修改了项目的源码，如何做？

  1. 重新构建image
  2. 删除旧容器
  3. 启动新容器

3.2 Replace the old container

  • 首先通过docker ps来获取容器的ID
  • 然后利用docker stop命令来停止容器
  • 一旦容器停止了就可以用docker rm命令将它移除了。

也可以在一条命令中停止和移除一个容器：

docker rm -f <the-container-id>

4. Share the application

如何与他人共享我们的镜像？

  • 首先需要一个Docker registry，默认是Docker Hub。
  • 创建repository
  • 推送image到这个repository，但需要首先为它命名
    • docker login -u YOUR-USER-NAME，首先登录到Docker Hub
    • docker tag getting-started YOUR-USER-NAME/getting-started，为该镜像提供一个新的名字，如果你没有指定一个标签，Docker会使用一个叫做latest的标签
    • docker push YOUR-USER-NAME/getting-started，推送这个image

然后就可以在别的机器上运行你的容器了，docker run -dp 3000:3000 YOUR-USER-NAME/getting-started。

5. Persist the DB

如何保证数据持久性？

每个容器每次启动时都是从镜像定义开始的。虽然容器可以创建、更新和删除文件，但当容器被删除时，这些变化就会丢失，所有的变化都被隔离在该容器之外。

通过volume，我们可以改变这一切。volume提供了将容器的特定文件系统路径连接回主机的能力。如果容器中的一个目录被挂载，该目录的变化也会在主机上看到。如果我们跨容器重启时挂载同一个目录，我们会看到相同的文件。

一种类型的volume叫：named volume。

通过创建一个卷，并将其附加（通常称为 “挂载”，mount）到数据存储的目录上，我们就可以持久化数据。

docker volume create todo-db，创建一个名为todo-db的volume

重新启动容器，但添加-v标志以指定一个卷的挂载。我们将使用命名的卷，并将其挂载到/etc/todos，这将捕获在该路径下创建的所有文件：

docker run -dp 3000:3000 -v todo-db:/etc/todos getting-started

这样就可以持久化数据了。

另外，可以通过docker volume inspect todo-db，查看docker实际上是在哪里持久化数据的。

6. Use bind mounts

docker所支持的另一种类型的卷，bind mounts，通过它我们可以控制主机上的确切安装点，通常用它向容器中提供额外的数据（例如源代码，如果我们修改了源代码，它能让我们实时看到修改）。

docker run -dp 3000:3000 \
     -w /app -v "$(pwd):/app" \
     node:12-alpine \
     sh -c "yarn install && yarn run dev"
  • -w /app 设置 “工作目录 “或当前目录，命令将从该目录运行
  • -v "$(pwd):/app" 绑定挂载容器中的主机的当前目录到/app目录中
  • node:12-alpine 指定要使用的image
  • sh -c "yarn install && yarn run dev"，启动一个shell，并运行命令

使用bind mounts在本地开发设置中是非常常见的。其优点是，开发机器不需要安装所有的构建工具和环境。只需一条docker run命令，开发环境就会被拉动并准备就绪。

我们将在未来的步骤中谈论Docker Compose，因为这将有助于简化我们的命令。

7. Multi container apps

如果两个容器在同一个网络上，那么它们可以相互访问，如果不在那么就不行。

7.1 设置网络

docker network create todo-app，创建网络

启动一个MySQL容器并将其连接到网络上，我们还将定义一些环境变量，数据库将使用这些变量来初始化数据库。

docker run -d \
     --network todo-app --network-alias mysql \
     -v todo-mysql-data:/var/lib/mysql \
     -e MYSQL_ROOT_PASSWORD=secret \
     -e MYSQL_DATABASE=todos \
     mysql:5.7

访问数据库：

docker exec -it <mysql-container-id> mysql -u root -p

7.2 连接到MySQL

nicolaka/netshoot容器带有很多工具，对排除故障或调试网络问题很有用。

docker run -it --network todo-app nicolaka/netshoot
dig mysql

docker能够将其解析为具有该网络别名的容器的IP地址（还记得我们之前使用的--network-alias标志吗）。

7.3 使用MySQL运行APP

MySQL 8.0以上需要运行以下命令：

ALTER USER 'root' IDENTIFIED WITH mysql_native_password BY 'secret';
flush privileges;

将应用指定到我们的网络，并设置一些环境变量：

docker run -dp 3000:3000 \
   -w /app -v "$(pwd):/app" \
   --network todo-app \
   -e MYSQL_HOST=mysql \
   -e MYSQL_USER=root \
   -e MYSQL_PASSWORD=secret \
   -e MYSQL_DB=todos \
   node:12-alpine \
   sh -c "yarn install && yarn run dev"

8. Use Docker Compose

Docker Compose是一个帮助定义和分享多容器应用程序的工具。使用Compose，我们可以创建一个YAML文件来定义服务，只需一个命令，就可以将所有的东西启动或关闭。

8.1 创建Compose文件

version: "3.7" # start off by defining the schema version

services: # 定义我们希望作为应用程序的一部分运行的服务（或容器）的列表。
  app: # 我们可以为该服务选择任何名字。这个名字将自动成为一个网络别名，这在定义我们的MySQL服务时将很有用。
    image: node:12-alpine
    command: sh -c "yarn install && yarn run dev"
    ports:
      - 3000:3000
    working_dir: /app
    volumes:
      - ./:/app
    environment:
      MYSQL_HOST: mysql
      MYSQL_USER: root
      MYSQL_PASSWORD: secret
      MYSQL_DB: todos
  mysql:
    image: mysql:5.7.35
    volumes: 
      - todo-mysql-data:/var/lib/mysql
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: todos
  
volumes:
  todo-mysql-data:

8.2 运行这组服务

docker-compose up -d

默认情况下，Docker Compose会自动为应用程序栈创建一个网络（这就是为什么我们没有在compose文件中定义一个网络）。

可以用docker-compose logs -f命令来看看日志。每个服务的日志都交错成一个流。-f标志是 “跟随 “日志的，所以会在日志产生时给你实时输出。

如果你想查看某个特定服务的日志，你可以在日志命令的末尾加上服务名称（例如，docker-compose logs -f app）。

8.3 停止这组服务

docker-compose down

容器将停止，网络也将被移除。

默认情况下，在运行docker-compose down时，不会删除compose文件中命名的卷。如果你想移除卷，你需要添加 --volumes标志。

9. Image-building best practices

9.1 Image layering

可以看到用于创建图像中每一层的命令。

docker image history getting-started

查看完整输出：

docker image history --no-trunc getting-started

9.2 Layer caching

需要记住一个关于image构建时间的重要原则：一旦一个层发生变化，所有下游的层也必须重新创建。

# syntax=docker/dockerfile:1
FROM node:12-alpine
WORKDIR /app
COPY . .
RUN yarn install --production
CMD ["node", "src/index.js"]

Dockerfile中的每条命令都成为image中的一个新层。也就是说如果修改了镜像，那么后边yarn的依赖关系必须重新安装。

为了解决这个问题，我们需要重组我们的Dockerfile，以帮助支持对依赖关系的缓存。

那么，如果我们先只复制该文件，安装依赖关系，然后再复制其他东西，会怎么样呢？

# syntax=docker/dockerfile:1
FROM node:12-alpine
WORKDIR /app
COPY package.json yarn.lock ./ # 
RUN yarn install --production
COPY . .
CMD ["node", "src/index.js"]

需要创建一个.dockerignore文件：

node_modules

.dockerignore文件是一种简单的方法，可以有选择地只复制与image相关的文件。在这种情况下，node_modules文件夹应该在第二个COPY步骤中被省略，因为否则，它可能会覆盖由RUN步骤中的命令创建的文件。

然后构建它：

docker build -t getting-started .

之后，如果修改了html文件，再重新构建时，前边的步骤就能够使用缓存了，所以重新构建的速度非常快。

9.3 Multi-stage builds

可以将构建时的依赖性与运行时的依赖性分开，通过只发送你的应用程序需要运行的内容来减少整体image大小。

1. Maven/Tomcat example

我们使用一个阶段（称为build），用Maven进行实际的Java构建。

在第二阶段（从FROM tomcat开始），我们把构建阶段的文件复制进来。最后的image只是创建的最后一个阶段（可以用-target标志覆盖）。

# syntax=docker/dockerfile:1
FROM maven AS build
WORKDIR /app
COPY . .
RUN mvn package

FROM tomcat
COPY --from=build /app/target/file.war /usr/local/tomcat/webapps

2. React example

这里，我们使用一个node来执行构建(最大化层缓存)，然后将输出复制到nginx容器中。

# syntax=docker/dockerfile:1
FROM node:12 AS build
WORKDIR /app
COPY package* yarn.lock ./
RUN yarn install
COPY public ./public
COPY src ./src
RUN yarn run build

FROM nginx:alpine
COPY --from=build /app/build /usr/share/nginx/html
